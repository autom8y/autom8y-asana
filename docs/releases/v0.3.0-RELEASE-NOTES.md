# Release Notes: autom8_asana v0.3.0

**Release Date**: 2025-12-12
**Status**: Ready for Production
**Type**: Major Release (Breaking Changes)

## Overview

v0.3.0 is a major release focused on **UX Remediation** - making the SDK fail-fast with clear error messages instead of silent failures. This release includes breaking changes justified by significantly improved user experience and prevention of silent errors.

**Key Improvement**: Users now get actionable error messages immediately at the call site, rather than cryptic failures from the Asana API.

---

## What's New

### Critical UX Fixes (5 Issues Resolved)

This release fixes all 5 critical UX issues identified in the QA post-mortem of v0.2.0:

#### Issue #1: CustomFieldAccessor Name Resolution (CRITICAL)
**Problem**: Unknown field names silently failed at API time with cryptic errors.
**Solution**: Strict mode (default) validates field names immediately with helpful suggestions.

**Before v0.3.0**:
```python
task = await client.tasks.get_async("1234567890")
task.custom_fields["UnknownField"] = "value"
await session.commit_async()  # ❌ Fails at API with vague error
```

**After v0.3.0**:
```python
task = await client.tasks.get_async("1234567890")
task.custom_fields["UnknownField"] = "value"  # ❌ Raises NameNotFoundError immediately!
# Error message:
# NameNotFoundError: custom_field 'UnknownField' not found.
# Did you mean: Priority, Status, Team?
# Available fields: Budget, Category, Priority, Status, Team
```

**Breaking Change**: `strict=True` is now default. Unknown field names raise `NameNotFoundError` instead of silently passing through.

**Escape Hatch**: Use `strict=False` for backward compatibility:
```python
task.custom_fields = CustomFieldAccessor(data, strict=False)  # Old behavior
```

---

#### Issue #2: SaveSession Error Messages (HIGH)
**Problem**: Only first error shown; other failures invisible.
**Solution**: All failures collected and shown together in `SaveSessionError`.

**Before v0.3.0**:
```python
async with SaveSession(client) as session:
    session.track(task1)
    task1.name = "Updated"
    session.track(task2)
    task2.name = "Also Updated"
    result = await session.commit_async()
# If 2 failures occur: only sees first failure ❌
```

**After v0.3.0**:
```python
async with SaveSession(client) as session:
    # ... same code ...
    result = await session.commit_async()
# If 2 failures occur: sees ALL failures ✓
# Accessible via: result.failed  (list of all failed operations)
```

**Breaking Change**: Exception type for save errors changed from individual operation error to `SaveSessionError`.

**Migration**:
```python
# Before:
try:
    await session.commit_async()
except SomeOperationError as e:
    # Caught first error only
    pass

# After:
try:
    await session.commit_async()
except SaveSessionError as e:
    for failed_op in e.result.failed:
        print(f"Operation {failed_op.operation_type} failed: {failed_op.error}")
```

**Backward Compatibility**: `SaveSessionError` extends `SaveOrchestrationError` (parent class). Existing `except SaveOrchestrationError` catches still work.

---

#### Issue #3: Custom Field Type Validation (HIGH)
**Problem**: Invalid types accepted at save time; fail at API.
**Solution**: Type validation at `set()` time with clear error messages.

**Before v0.3.0**:
```python
task = await client.tasks.get_async("1234567890")
task.custom_fields.set("Priority", ["high"])  # Wrong type (list instead of str)
await session.commit_async()  # ❌ Fails at API with vague error
```

**After v0.3.0**:
```python
task = await client.tasks.get_async("1234567890")
task.custom_fields.set("Priority", ["high"])
# ❌ Raises ValidationError immediately:
# ValidationError: Custom field 'Priority' expects enum (str), got list
```

**New Feature**: Support for `decimal.Decimal` in number fields for financial precision:
```python
from decimal import Decimal
task.custom_fields.set("Budget", Decimal("1234.56"))  # ✓ Now supported
```

**Breaking Change**: Previously-accepted invalid types now raise `ValidationError`.

**Migration**: Fix the type of values being set:
```python
# Before (worked by accident):
task.custom_fields.set("Priority", ["high"])  # Wrong type

# After (correct):
task.custom_fields.set("Priority", "high")  # Correct type
```

---

#### Issue #4: SaveSession Inspect Methods (HIGH)
**Status**: ✓ **Already implemented in v0.2.0**

No changes needed. The following methods are available:
- `get_pending_cascades()` - List cascading operations
- `get_pending_actions()` - List action operations
- `preview()` - Get preview of all operations
- `get_dirty_entities()` - Get entities with changes

---

#### Issue #5: GID Input Validation (HIGH)
**Problem**: Invalid GIDs passed to methods, fail at API.
**Solution**: Validate GID format at method entry with clear error messages.

**Before v0.3.0**:
```python
await client.tasks.add_tag_async("abc-invalid", tag_gid="9876543210")
# ❌ Fails at API with generic error
```

**After v0.3.0**:
```python
await client.tasks.add_tag_async("abc-invalid", tag_gid="9876543210")
# ❌ Raises ValidationError immediately:
# ValidationError: Invalid task_gid: 'abc-invalid'.
# GIDs must be non-empty numeric strings (1-64 digits).
```

**Breaking Change**: Malformed GIDs now raise `ValidationError` immediately.

**Migration**: Use valid numeric GIDs:
```python
# Before (worked by luck):
await client.tasks.add_tag_async("task-123", tag_gid="tag-456")  # Wrong format

# After (correct):
await client.tasks.add_tag_async("1234567890", tag_gid="9876543210")  # Correct
```

## Breaking Changes Summary

| Issue | Old Behavior | New Behavior | Why It's Better |
|-------|--------------|--------------|-----------------|
| #1: Field Names | Unknown fields pass through, fail at API | Unknown fields raise `NameNotFoundError` immediately | Fail-fast, actionable error message |
| #2: Error Messages | Only first error shown | All errors collected in `SaveSessionError` | Users see all failures, not just first |
| #3: Type Validation | Invalid types accepted, fail at API | Invalid types raise `ValidationError` immediately | Fail-fast, clear type error message |
| #5: GID Validation | Invalid GIDs pass through, fail at API | Invalid GIDs raise `ValidationError` immediately | Fail-fast, clear format error message |

## Migration Guide

### For Issue #1: Field Name Resolution

**If your code sets custom fields by name:**

```python
# Check what changed:
# - Unknown field names now raise NameNotFoundError
# - Suggestions provided to help fix typos

# If you get NameNotFoundError:
# 1. Check the suggested field names
# 2. Fix the typo in your code
# 3. Or use explicit field GID instead of name

# Option A: Fix the field name
task.custom_fields["Priority"] = "High"  # Correct spelling

# Option B: Use explicit field GID (more reliable)
PRIORITY_FIELD_GID = "1234567890"  # Get from Asana workspace
task.custom_fields.set(PRIORITY_FIELD_GID, "High")

# Option C: Disable strict mode (not recommended)
task.custom_fields = CustomFieldAccessor(data, strict=False)  # Legacy behavior
```

### For Issue #2: SaveSession Error Handling

**If your code handles save errors:**

```python
# Before v0.3.0:
try:
    result = await session.commit_async()
except AsanaError as e:
    print(f"Failed: {e}")
    # Only saw first error

# After v0.3.0:
try:
    result = await session.commit_async()
except SaveSessionError as e:
    # See ALL failures
    for failed_op in e.result.failed:
        print(f"Operation failed: {failed_op}")
        print(f"  Type: {failed_op.operation_type}")
        print(f"  Error: {failed_op.error}")
        print(f"  Details: {failed_op.error_details}")
except SaveOrchestrationError as e:
    # Still catches SaveSessionError (parent class)
    # Use this if you want broader error handling
    pass
```

### For Issue #3: Custom Field Types

**If your code sets custom field values:**

```python
# Check your field assignments for type errors:

# WRONG: Lists on enum fields
task.custom_fields["Status"] = ["Active"]  # ❌ List instead of str

# CORRECT: Scalar values for enum
task.custom_fields["Status"] = "Active"  # ✓ String value

# WRONG: Strings on number fields
task.custom_fields["Budget"] = "1000.50"  # ❌ String instead of number

# CORRECT: Numeric values for number fields
task.custom_fields["Budget"] = 1000.50  # ✓ Float
task.custom_fields["Budget"] = Decimal("1000.50")  # ✓ Decimal (new!)

# WRONG: Single values on multi-enum
task.custom_fields["Skills"] = "Python"  # ❌ String instead of list

# CORRECT: Lists for multi-enum
task.custom_fields["Skills"] = ["Python", "Go"]  # ✓ List of strings
```

### For Issue #5: GID Format

**If your code passes GIDs to methods:**

```python
# WRONG: Non-numeric GID format
await client.tasks.add_tag_async("task-123", tag_gid="tag-456")  # ❌

# CORRECT: Numeric GID format
await client.tasks.add_tag_async("1234567890", tag_gid="9876543210")  # ✓

# Get real GIDs from Asana API:
project = await client.projects.get_async("1234567890")
# project.gid is numeric (e.g., "1234567890")

# If you have mock/test GIDs:
# Replace non-numeric patterns like "task-123" with numeric "1234567890"
```

## Backward Compatibility

### What Still Works

✓ All public API method signatures are unchanged
✓ SaveSession context manager works exactly the same
✓ CustomFieldAccessor dictionary interface unchanged
✓ Exception hierarchy preserved (new exceptions extend old ones)
✓ Existing async/sync patterns work unchanged
✓ Rate limiting, retries, logging unchanged

### What Changed

❌ **New Exceptions Raised** in scenarios that previously failed silently
- `NameNotFoundError` - Unknown custom field names (strict mode)
- `ValidationError` - Invalid GIDs or field types
- `SaveSessionError` - Replaces individual operation errors (but extends parent)

❌ **CustomFieldAccessor strict parameter** (default `True`, can set `False` for legacy)
❌ **SaveSession error collection** (now collects all failures, not just first)

### Compatibility Strategy

1. **Exception Hierarchy**: New exceptions extend existing parents
   ```python
   # This still catches all errors:
   except AsanaError:  # Parent class still works
       pass

   # More specific:
   except NameNotFoundError:  # New exception type
       pass
   ```

2. **Strict Mode Escape Hatch**: Use `strict=False` for legacy behavior
   ```python
   accessor = CustomFieldAccessor(data, strict=False)
   accessor["UnknownField"] = "value"  # Returns as-is (old behavior)
   ```

3. **Error Message Changes**: Some error messages now more detailed (improvement)

## Known Limitations

1. **Strict Mode Field Names**: Only suggests names from current task's custom fields. For workspace-wide suggestions, use explicit GID.

2. **Custom Field Validation**: Validates type but not enum values. Invalid enum values still fail at API time.

3. **Cross-Workspace GIDs**: SDK assumes all GIDs are in same workspace context. Cross-workspace operations not supported by Asana API.

4. **Temporary GID Format**: `temp_` prefix pattern not enforced at validation level (by design).
