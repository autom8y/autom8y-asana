# ADR-0034: HTTP Transport & SDK Integration Strategy

## Metadata
- **Status**: Accepted
- **Author**: Architect
- **Date**: 2025-12-25
- **Deciders**: Architect, Principal Engineer
- **Consolidated From**: ADR-0003 (HTTP Layer), ADR-0007 (Client Pattern), ADR-0013 (Correlation IDs), ADR-0063 (Client References)
- **Related**: reference/API-INTEGRATION.md

## Context

The SDK requires async-first HTTP operations with custom rate limiting, retry logic, and concurrency control. The official Asana Python SDK provides type definitions and error parsing patterns but uses sync-only HTTP transport incompatible with our performance requirements.

Key requirements:
- Async operations (not sync wrapped in threads)
- Custom rate limiting (token bucket at 1500 req/min)
- Exponential backoff with jitter
- Connection pooling configuration
- Consistent client patterns across all resource types
- Request correlation for debugging

The official `asana` SDK (5.0.3+) provides valuable type definitions and battle-tested error parsing but cannot be used directly for HTTP transport.

## Decision

**Replace Asana SDK's HTTP layer with httpx-based async transport while retaining SDK types for reference and compatibility verification.**

### Architecture

```
┌─────────────────────────────────────────────────┐
│              AsanaClient                        │
│  ├── tasks: TasksClient                        │
│  ├── projects: ProjectsClient                  │
│  ├── users: UsersClient                        │
│  └── [resource clients...]                     │
└─────────────────────────────────────────────────┘
                      │
                      v
┌─────────────────────────────────────────────────┐
│         AsyncHTTPClient (httpx)                 │
│  ├── TokenBucketRateLimiter                    │
│  ├── RetryHandler (exponential backoff)        │
│  ├── Connection Pooling                        │
│  └── Correlation ID Generation                 │
└─────────────────────────────────────────────────┘
```

### Client Pattern

All resource clients follow consistent pattern:

```python
class TasksClient:
    def __init__(self, http_client: AsyncHTTPClient):
        self._http = http_client

    async def get_async(
        self,
        task_gid: str,
        *,
        opt_fields: list[str] | None = None,
        raw: Literal[False] = False,
    ) -> Task: ...

    async def get_async(
        self,
        task_gid: str,
        *,
        opt_fields: list[str] | None = None,
        raw: Literal[True],
    ) -> dict[str, Any]: ...

    def list_async(
        self,
        *,
        workspace: str | None = None,
        project: str | None = None,
        opt_fields: list[str] | None = None,
    ) -> PageIterator[Task]:
        """Return async iterator for paginated results."""
```

Key attributes:
- **Async-first**: All I/O operations return coroutines
- **Overloaded signatures**: `raw=True` returns dict, `raw=False` returns Pydantic model
- **PageIterator for lists**: Lazy pagination, fetches on iteration
- **Consistent parameters**: `opt_fields` for field selection, `raw` for response format

### Correlation ID Strategy

Generate SDK-scoped correlation IDs for request tracing:

**Format**: `sdk-{timestamp_hex}-{random_hex}` (18 characters)
- Timestamp: Lower 32 bits of Unix timestamp (milliseconds) as 8 hex chars
- Random: 16 bits of randomness as 4 hex chars

**Scope**: Per-operation (one ID covers all HTTP requests for a single SDK call)

**Propagation**: Decorator-based injection via `@error_handler`

```python
@error_handler
async def get_async(self, task_gid: str, ...) -> Task:
    # Correlation ID generated by decorator
    # Available in logs and exceptions
```

### Client Reference Storage

Task convenience methods require client access:

```python
class Task(BaseModel):
    _client: AsanaClient | None = PrivateAttr(default=None)

    async def save_async(self) -> Task:
        """Convenience method using stored client."""
        if self._client is None:
            raise ConfigurationError("Task not associated with client")
        return await self._client.tasks.update_async(self.gid, self)
```

**Storage**: Strong reference (not WeakReference) in `PrivateAttr`
- Simple O(1) access
- No dereferencing overhead
- One-way link (no circular reference issues)
- Tasks are short-lived; memory impact acceptable

## Rationale

### Why Custom HTTP Layer?

Official SDK limitations:
- Sync-only (no native async support)
- No custom rate limiting
- No configurable retry strategies
- No connection pooling configuration

Custom httpx transport provides:
- Native async operations
- Token bucket rate limiting (1500 req/min)
- Exponential backoff with jitter
- Connection pool configuration
- Request/response middleware hooks

### Why Keep Asana SDK Dependency?

Retaining `asana` package provides:
1. **Type reference**: Verify our Pydantic models match Asana's expectations
2. **Error patterns**: Battle-tested error parsing logic
3. **API changes**: Compare against SDK updates to detect API drift
4. **Compatibility**: Satisfy dependency requirements

### Why Correlation IDs?

Asana's `X-Request-Id` is insufficient:
- Only available after request completes
- Changes on retry
- Multiple IDs for paginated operations
- Not available when request fails before reaching API

SDK-generated IDs provide:
- Consistent tracing across all HTTP activity for a single operation
- Available before any request
- Temporal ordering via timestamp prefix
- Low collision risk (~1/65536 per millisecond)

### Why Decorator-Based Propagation?

Alternatives considered:
- **Class decorator**: Implicit, wraps internal methods unnecessarily
- **`__getattribute__` magic**: Clever but hard to debug
- **Manual context in each method**: Verbose, error-prone
- **Contextvars**: Magic implicit state, harder to test

Explicit `@error_handler` decorator:
- Clear boundaries for error handling
- Standard Python pattern
- Works with type checkers
- Easy to understand and test

## Alternatives Considered

### Full Adoption (Use Asana SDK Directly)

- **Pros**: Less code, automatic updates, battle-tested
- **Cons**: Sync-only, no custom rate limiting/retry, cannot meet async requirements
- **Why not chosen**: Official SDK doesn't support async-first design

### Complete Replacement (No Asana SDK)

- **Pros**: Complete independence, no unused dependencies
- **Cons**: Must track API changes manually, no reference implementation, violates dependency requirement
- **Why not chosen**: Keeping SDK provides valuable type reference

### Fork Asana SDK

- **Pros**: Start with working code, incremental async addition
- **Cons**: Maintenance burden, license considerations, significant refactoring needed
- **Why not chosen**: Forks are maintenance nightmares; clean rebuild is better

### Async Wrapper Around Sync SDK

- **Pros**: Quick implementation, uses battle-tested SDK
- **Cons**: Thread overhead per call, doesn't solve rate limiting/retry, defeats async purpose
- **Why not chosen**: Thread-per-call defeats async benefits

### WeakReference for Client Storage

- **Pros**: No strong reference from task to client
- **Cons**: Dereferencing overhead, added complexity, client outlives task in typical use
- **Why not chosen**: Strong reference is simpler and sufficient

## Consequences

### Positive

- **Full async support**: Native httpx async HTTP
- **Custom transport control**: Exact rate limiting, retry, and concurrency behavior needed
- **Connection pooling**: Efficient resource usage
- **Type safety**: Reference Asana types for compatibility verification
- **Request tracing**: Every operation traceable via correlation ID
- **Consistent patterns**: All clients follow same structure
- **Convenience methods**: Tasks can save themselves via stored client

### Negative

- **More code to maintain**: Own the transport layer
- **Potential drift**: Pydantic models might diverge from Asana types over time
- **Dependency not fully utilized**: Include `asana` but only use types/errors
- **Duplicate type definitions**: Pydantic models overlap with SDK classes
- **Collision risk**: Small chance of correlation ID collision (~1/65536 per ms)

### Neutral

- **Two type systems**: Asana SDK classes and our Pydantic models
- **Error hierarchy**: Own exceptions but reference Asana patterns
- **Import auditing needed**: Verify `asana` imports only for types/errors

## Compliance

### Enforcement

1. **No direct Asana SDK HTTP calls**: Code review blocks `asana.Client` usage
2. **Import auditing**: CI checks that `asana` imports are types/errors only
3. **Architecture tests**: Verify httpx is only HTTP library making requests
4. **Client pattern compliance**: All resource clients follow TasksClient pattern
5. **Correlation ID presence**: Verify `@error_handler` on all public async methods

### Validation

- [ ] All HTTP requests use `AsyncHTTPClient`, not Asana SDK transport
- [ ] All resource clients inherit from `BaseClient` with consistent patterns
- [ ] Correlation IDs generated for all operations
- [ ] Correlation IDs included in logs and exceptions
- [ ] Client references stored in tasks when created via client
- [ ] Documentation explains relationship with official SDK
