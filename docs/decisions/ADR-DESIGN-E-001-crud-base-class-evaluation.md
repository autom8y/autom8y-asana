# ADR-DESIGN-E-001: CRUD Client Base Class Evaluation

| Field | Value |
|-------|-------|
| **ID** | ADR-DESIGN-E-001 |
| **Title** | CRUD Client Base Class Evaluation |
| **Status** | Accepted |
| **Date** | 2025-12-16 |
| **Context** | Design Patterns Sprint - Initiative E |

---

## Context

The Design Patterns Sprint meta-initiative identified CRUD client duplication as a pattern opportunity. Initiative E proposed a `CRUDClientMeta` metaclass that would auto-generate CRUD methods, with an estimated savings of ~500+ lines across clients.

Discovery and technical analysis was conducted to assess feasibility.

## Decision

**DO NOT implement CRUDClient base class or CRUDClientMeta metaclass.**

The analysis revealed that:
1. @async_method from Initiative D already captured 65% of the boilerplate
2. Type overloads cannot be runtime-generated for mypy/IDE support
3. Create method signatures vary too much between resources
4. Net benefit after accounting for base class complexity is negative or negligible

## Analysis

### What the Meta-Initiative Assumed

| Assumption | Reality |
|------------|---------|
| Type overloads can be generated at runtime | No - mypy reads source code, not runtime |
| Create methods can be templated | No - each resource has unique required parameters |
| Major boilerplate remains after @async_method | No - @async_method eliminated 65% |
| 500+ lines can be saved | No - ~10% per client after @async_method |

### Line Count Analysis

**Current State (with @async_method)**:
- SectionsClient: ~156 lines
- TagsClient: ~350 lines (uses old pattern)
- ProjectsClient: ~450 lines (uses old pattern)

**Projected State (with CRUDClient)**:
- SectionsClient: ~140 lines (saves 16)
- CRUDClient base: ~80 lines (new code)
- Net change for one client: -64 lines (worse)
- Break-even: 5+ clients migrated

**Actual Value**: Minimal savings that don't justify the abstraction.

### Python Type System Constraints

The `raw` parameter changes return type (`T` vs `dict`). This requires `@overload` declarations:

```python
@overload
async def get_async(self, gid: str, *, raw: Literal[False] = ...) -> T: ...
@overload
async def get_async(self, gid: str, *, raw: Literal[True]) -> dict: ...
```

These MUST be in source code for type checkers. They cannot be:
- Generated by a metaclass
- Generated by a decorator
- Placed in .pyi stubs only (would break IDE experience)

This is a **fundamental Python limitation**, not a design choice.

### Create Method Variability

Each resource has unique create parameters:

| Resource | Required Parameters |
|----------|---------------------|
| Task | name, (workspace OR projects OR parent) |
| Section | name, project |
| Tag | workspace, name |
| Project | name, workspace |

A generic `create()` template cannot handle this variation without becoming more complex than individual implementations.

## Consequences

### Positive

1. **No new abstraction layer** - codebase remains simpler
2. **@async_method pattern is sufficient** - validated as right level of abstraction
3. **Documentation captures learnings** - future decisions benefit from this analysis

### Negative

1. **Some duplication remains** - type overloads cannot be eliminated
2. **Meta-initiative scope was optimistic** - Initiative E delivers analysis, not implementation

### Neutral

1. **Other clients can adopt @async_method** - per Initiative D migration guide
2. **Pattern documented** - code conventions can reference this ADR

## Alternatives Considered

### 1. Metaclass (Original Proposal)

**Description**: `CRUDClientMeta` generates methods at class creation time.

**Rejected Because**:
- Type overloads cannot be generated for static analysis
- High complexity, mypy plugin would be needed
- Breaks IDE autocomplete

### 2. Generic Base Class (Analyzed)

**Description**: `CRUDClient[T]` provides template methods that subclasses delegate to.

**Rejected Because**:
- Overloads still required in subclasses
- Create cannot be templated
- ~10% savings doesn't justify abstraction

### 3. Protocol Mixin

**Description**: Mixin providing helper methods without inheritance hierarchy.

**Not Recommended Because**:
- Even less savings than base class
- Adds complexity without proportional benefit

### 4. Current Pattern with @async_method (Chosen)

**Description**: Keep independent clients using @async_method decorator.

**Chosen Because**:
- Simple and maintainable
- @async_method already handles major duplication
- No new abstraction to learn
- Each client is self-contained
