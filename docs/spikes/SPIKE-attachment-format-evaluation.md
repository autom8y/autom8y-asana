## Question

What file format should `insights_export` use for data attachments uploaded to Asana tasks? The current markdown (pipe-table) approach produces unreadable output because Asana does NOT render markdown -- it shows raw pipe-delimited text. The legacy `.txt` approach using Python's `tabulate` library with aligned columns is dramatically more legible.

Secondary: Should `conversation_audit` (currently CSV) also change?

## Decision This Informs

Whether to:
- (A) Revert `insights_export` to tabulate-formatted `.txt` (legacy approach)
- (B) Switch to a different format (HTML, CSV, TSV, XLSX)
- (C) Use a hybrid approach (narrative text + tabulate data tables)
- (D) Keep markdown `.md` as-is

---

## Findings

### 1. Asana Attachment Preview Behavior

Asana's documentation does not publish a definitive list of formats that receive inline preview treatment. Based on research across official docs, community forums, API documentation, and third-party analysis, the observed behavior is:

| Format | Asana Preview Behavior | Evidence Source |
|--------|----------------------|----------------|
| **Images** (PNG, JPG, GIF) | Full inline preview with lightbox | Official docs, universal user reports |
| **PDF** | Inline preview (multi-page viewer) | Official docs, user reports |
| **Video** (MP4) | Inline player | User reports |
| **Google Drive** links | Inline preview via Drive integration | Official docs |
| **Office docs** (DOCX, XLSX, PPTX) | Thumbnail preview, click to open | User reports (varies) |
| **.txt** | **Download only** -- no inline render | Empirical (your legacy workflow) |
| **.csv** | **Download only** -- no inline render | Empirical (conversation_audit workflow) |
| **.md** | **Download only** -- no inline render, raw text if opened | Empirical (current insights_export) |
| **.html** | **Unknown** -- likely download only (not a documented preview type) | No evidence of inline HTML render |
| **.tsv** | **Download only** -- same as CSV | No evidence |

**Critical finding**: None of the text-based formats (TXT, CSV, MD, HTML, TSV) get inline preview in Asana. All require the user to download and open locally. This means the evaluation criteria should weight **"download legibility"** (what happens when the user opens the file) higher than originally expected, and **"Asana preview legibility"** is really about the *filename and file size visible in the attachment list*, not about rendered content.

The real legibility question is: **When the user downloads and opens this file, how readable is it?**

### 2. Current Architecture

Two workflows produce attachments:

**insights_export** (the problem case):
- File: `insights_export_{business}_{date}.md` (~462KB, ~1039 lines)
- Content type: `text/markdown`
- Structure: Header + 10 pipe-delimited tables + footer
- Generated by: `insights_formatter.py` (274 lines, pure functions)
- Upload: `AttachmentsClient.upload_async()` with `content_type="text/markdown"`

**conversation_audit** (working fine):
- File: `conversations_{phone}_{date}.csv` (~variable)
- Content type: `text/csv`
- Structure: Standard CSV from `DataServiceClient.get_export_csv_async()`
- Generated by: External service (autom8_data), consumed as bytes

### 3. Why Markdown is 4x Larger and Unreadable

The current `insights_formatter.py` produces pipe-table markdown:

```
| Office Phone | Vertical | Office | Spend | Leads | Scheds | ... 14 columns |
| --- | --- | --- | --- | --- | --- | --- |
| +16785130095 | chiropractic | Natural Medicine... | 322.42 | 3 | 0 | ... |
```

Problems:
1. **No column alignment** -- pipe tables have variable-width cells
2. **Redundant metadata** -- phone, vertical, office repeated on EVERY row (already in header)
3. **Pipe noise** -- `|` characters everywhere add visual clutter
4. **Separator row** -- `| --- |` consumes a full line per table for zero value
5. **462KB vs 107KB** -- markdown overhead and redundant data inflate size 4x

The legacy tabulate approach:

```
  num_pmts    weekly_budget  pmts_start    pmts_end
----------  ---------------  ------------  ----------
        80              400  2024-02-12    2025-11-17
```

Advantages: Fixed-width columns, right-aligned numbers, no redundant data, compact.

### 4. Tabulate Library Capabilities

The `tabulate` library (PyPI) supports 35+ output formats including:

| Format | tablefmt Value | Characteristics |
|--------|---------------|-----------------|
| **simple** | `"simple"` | Minimal separators, clean |
| **grid** | `"grid"` | Full box-drawing borders |
| **github** | `"github"` | GitHub-flavored markdown tables |
| **pipe** | `"pipe"` | Current broken approach (pipe-delimited) |
| **plain** | `"plain"` | No separators at all, just aligned columns |
| **pretty** | `"pretty"` | ASCII art borders, wide |
| **html** | `"html"` | Full `<table>` with `<th>`/`<td>` tags |
| **tsv** | `"tsv"` | Tab-separated values |
| **rst** | `"rst"` | reStructuredText grid tables |

Key capabilities:
- Automatic column width calculation and alignment
- Right-aligns numeric columns by default
- `tablefmt="html"` produces `<table>` with inline `style` alignment attributes
- Does NOT support custom CSS injection natively (would require post-processing)
- Zero-dependency library (pure Python)
- Not currently in `pyproject.toml` -- would be a new dependency

### 5. Format-by-Format Analysis

#### 5a. Plain Text with Tabulate (.txt)

```
  num_pmts    weekly_budget  pmts_start    pmts_end
----------  ---------------  ------------  ----------
        80              400  2024-02-12    2025-11-17
```

- Requires monospace font to render correctly (TextEdit, VS Code, Terminal all use monospace for .txt)
- Legacy approach already proven in production (107KB, 330 lines)
- Handles narrative sections (headers, metadata) via plain text formatting
- Excellent number alignment (right-justified)

#### 5b. CSV (.csv)

```csv
num_pmts,weekly_budget,pmts_start,pmts_end
80,400,2024-02-12,2025-11-17
```

- Opens natively in Excel/Numbers/Google Sheets with column formatting
- But: all 10 tables would need to be in one file (no native multi-table CSV)
- Would require blank-row separators or sheet conventions between tables
- Loses narrative structure (header, footer, section titles)
- Conversation_audit already uses CSV for its single-table data -- appropriate there

#### 5c. HTML (.html)

```html
<table>
  <tr><th>num_pmts</th><th>weekly_budget</th></tr>
  <tr><td>80</td><td>400</td></tr>
</table>
```

- Opens in any browser with full table rendering
- Tabulate can generate HTML tables natively
- Inline CSS can add borders, zebra striping, header styling
- Supports narrative sections via `<h2>`, `<p>` tags
- But: Asana does NOT preview HTML inline -- user must download and open
- When opened locally: browser renders beautifully
- Risk: Some corporate environments block opening downloaded HTML files

#### 5d. XLSX / Excel (.xlsx)

- Multi-sheet workbook (one sheet per table) -- ideal for 10 tables
- Native formatting: number alignment, column widths, header styling
- Opens natively in Excel/Numbers/Google Sheets
- Requires `openpyxl` dependency (~15MB installed size)
- Binary format -- not diffable, not inspectable in text editors
- Most professional appearance for business users
- Highest implementation complexity

#### 5e. Markdown (.md) -- Current Approach

- As documented above: unreadable in Asana, unreadable when downloaded (raw pipes)
- Only advantage: renders on GitHub if the file were committed (irrelevant for Asana)
- Strictly worse than every alternative for this use case

#### 5f. Hybrid: Narrative Text + Tabulate Tables (.txt)

```
=== INSIGHTS EXPORT: Business Name ===
Phone: +1 (XXX) XXX-XXXX
Vertical: chiropractic
Generated: 2026-02-20T12:00:00Z

--- SUMMARY ---
  num_pmts    weekly_budget  pmts_start    pmts_end
----------  ---------------  ------------  ----------
        80              400  2024-02-12    2025-11-17

--- APPOINTMENTS (last 90 days, showing 100 of 247) ---
  date          name              status      phone
----------  ------------------  ----------  -----------
2026-02-19  John Smith          confirmed   +1XXX...0095
```

- Best of both worlds: structured narrative + legible data
- This is essentially what the legacy `.txt` format was
- Monospace rendering in any text editor
- Compact: ~107KB vs ~462KB

#### 5g. TSV (.tsv)

- Tab-separated aligns naturally in monospace viewers
- Same multi-table limitations as CSV
- Less widely recognized file extension than CSV
- No significant advantage over tabulate-formatted .txt

## Comparison Matrix

Scoring: 1 (worst) to 5 (best). Weight applied as multiplier.

| Criterion | Weight | TXT+tabulate | CSV | HTML | XLSX | Markdown | Hybrid TXT | TSV |
|-----------|--------|:---:|:---:|:---:|:---:|:---:|:---:|:---:|
| **Asana preview legibility** | 5x | 1 | 1 | 1 | 2 | 1 | 1 | 1 |
| **Download legibility** | 3x | 4 | 3 | 5 | 5 | 1 | 4 | 3 |
| **Implementation complexity** | 2x | 4 | 2 | 3 | 1 | 5 | 4 | 2 |
| **File size efficiency** | 1x | 4 | 5 | 3 | 3 | 1 | 4 | 5 |
| **Cross-workflow applicability** | 3x | 4 | 5 | 4 | 3 | 2 | 4 | 4 |
| **Weighted Total** | | **41** | **38** | **42** | **38** | **23** | **41** | **35** |

### Score Breakdown

**TXT+tabulate (41)**:
- Asana preview: 1x5=5 (download only, like all text formats)
- Download: 4x3=12 (great in monospace editors; not as polished as HTML/Excel)
- Complexity: 4x2=8 (tabulate is simple; new dep but zero-config)
- Size: 4x1=4 (compact aligned text)
- Cross-workflow: 4x3=12 (works for both; CSV better for conversation_audit specifically)

**HTML (42)**:
- Asana preview: 1x5=5 (download only; no inline HTML render)
- Download: 5x3=15 (beautiful in browser with styled tables)
- Complexity: 3x2=6 (tabulate HTML + CSS wrapper + multi-section template)
- Size: 3x1=3 (HTML tags add overhead)
- Cross-workflow: 4x3=12 (works for both; slight mismatch for raw CSV data)

**Hybrid TXT (41)**:
- Identical scoring to TXT+tabulate (same format, just clearer about the narrative structure)

**Markdown (23)**: Worst in every dimension except implementation complexity (already implemented).

### Scoring Notes

All text-based formats score identically on "Asana preview legibility" (1/5) because none get inline rendering. XLSX scores slightly better (2/5) because Asana may show a thumbnail or Excel icon that signals "open in spreadsheet app," but the content is still not inline.

The HTML vs TXT+tabulate decision comes down to:
- HTML wins on download legibility (browser rendering is polished)
- TXT wins on implementation simplicity (no HTML/CSS template needed)
- Both score the same overall

## Recommendation

**Adopt: Hybrid plain text with tabulate** (Option F / "Hybrid TXT"), implemented as the `"simple"` tablefmt in the tabulate library.

### Rationale

1. **No format gets inline Asana preview.** Since ALL text-based formats require download, the rendering engine is the user's local editor (TextEdit, VS Code, Notepad). Plain text with aligned columns is universally readable in every editor that defaults to monospace fonts -- which is all of them for `.txt` files.

2. **HTML is the highest-scoring alternative but has friction.** Users must download and double-click an HTML file, which opens a browser tab. Some corporate/security environments flag downloaded HTML. The marginal beauty improvement does not justify the workflow friction or the more complex generation template.

3. **Tabulate is battle-tested.** The legacy workflow already used this approach successfully in production. The 107KB/330-line legacy output vs. the 462KB/1039-line markdown output demonstrates the efficiency gain.

4. **Minimal implementation delta.** The change is confined to `insights_formatter.py` (274 lines). Replace pipe-table generation with `tabulate(data, headers="keys", tablefmt="simple")`. The narrative structure (header, section titles, footer) stays as plain text.

5. **Zero impact on conversation_audit.** CSV is the correct format for conversation_audit (single table, opens in Excel). No change needed there.

6. **Tabulate is a lightweight dependency.** Pure Python, no C extensions, ~50KB installed size. Well-maintained (6M+ monthly PyPI downloads).

### Confidence Level

**High (8/10).** The legacy approach was proven in production. The only uncertainty is whether `tabulate` column widths will handle the widest real-world data gracefully (some business names are long). This is easily validated with a dry-run test on production data.

### Why NOT HTML

HTML scored 42 vs 41, making it a marginal winner on points. However:
- The 1-point difference is within noise
- HTML requires a template with CSS (more code to maintain)
- HTML download-and-open is a less natural workflow than text file preview
- HTML files may trigger security warnings in some environments
- The tabulate library's `"html"` output has no built-in CSS -- we would need to wrap it in a full HTML document with inline styles

If user feedback indicates the text format is insufficiently polished for business stakeholders, HTML is the natural escalation path. The tabulate library supports `tablefmt="html"` so the migration would be a one-line change in the formatter plus an HTML document wrapper.

### Why NOT XLSX

XLSX would be the most professional format for business users and scored well on download legibility. However:
- `openpyxl` is a heavy dependency (~15MB) for a single use case
- Binary format makes debugging/testing harder
- Multi-sheet workbook generation is significantly more complex than text formatting
- We do not have evidence that stakeholders need Excel-specific features (formulas, sorting, filtering)

If stakeholders explicitly request Excel-compatible output, this is the correct escalation path.

## Migration Path

### Effort Estimate: 0.5-1 day

### Changes Required

**1. Add `tabulate` dependency** (`pyproject.toml`):
```toml
dependencies = [
    ...
    "tabulate>=0.9.0",
]
```

**2. Rewrite `insights_formatter.py`** (~274 lines, full rewrite of formatting functions):

Replace `_format_table_section()` pipe-table generation with tabulate:
```python
from tabulate import tabulate

def _format_table_section(table_name: str, rows: list[dict], row_limit: int | None = None) -> str:
    display_rows = rows[:row_limit] if row_limit else rows
    truncated = row_limit is not None and len(rows) > row_limit

    table_text = tabulate(display_rows, headers="keys", tablefmt="simple")

    parts = [f"--- {table_name} ---", "", table_text]
    if truncated:
        parts.extend(["", f"(Showing first {row_limit} of {len(rows)} rows)"])
    return "\n".join(parts)
```

Replace `_format_header()` markdown syntax with plain text:
```python
def _format_header(data: InsightsReportData) -> str:
    masked = mask_phone_number(data.office_phone)
    timestamp = datetime.now(UTC).isoformat()
    return (
        f"{'=' * 60}\n"
        f"  INSIGHTS EXPORT: {data.business_name}\n"
        f"{'=' * 60}\n"
        f"Phone:     {masked}\n"
        f"Vertical:  {data.vertical}\n"
        f"Generated: {timestamp}\n"
        f"Period:    Daily insights report"
    )
```

**3. Update `insights_export.py`**:
- Change filename extension: `.md` to `.txt`
- Change content_type: `"text/markdown"` to `"text/plain"`
- Change attachment pattern: `"insights_export_*.md"` to `"insights_export_*.txt"`

**4. Update tests** (`test_insights_formatter.py`, `test_insights_export.py`):
- Update expected output format from pipe tables to tabulate simple format
- Update filename/content_type assertions

**5. One-time cleanup**: The first run after deployment will upload new `.txt` files. The old `.md` files will NOT be cleaned up automatically because the attachment pattern changes from `*.md` to `*.txt`. Options:
- (a) Run one deployment with both patterns in cleanup: `["insights_export_*.md", "insights_export_*.txt"]`
- (b) Accept stale `.md` files until manual cleanup
- Recommendation: Option (a) -- add the old pattern as a transitional cleanup for one release cycle

### No Changes Required
- `conversation_audit.py` -- CSV format is appropriate, no change
- `AttachmentsClient` -- already supports arbitrary content types
- `mixins.py` -- attachment replacement mixin is format-agnostic
