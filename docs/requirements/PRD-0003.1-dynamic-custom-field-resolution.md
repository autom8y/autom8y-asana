## Problem Statement

The Phase 1 dataframes implementation contains a code smell that creates maintenance burden and limits workspace portability. ADR-0030 explicitly defers dynamic resolution to "post-MVP" - this PRD defines that enhancement.

**Current Implementation Issues**:

1. **16 Placeholder GID Constants** (`/src/autom8_asana/dataframes/models/custom_fields.py`):
   ```python
   MRR_GID: str = "PLACEHOLDER_MRR_GID"
   WEEKLY_AD_SPEND_GID: str = "PLACEHOLDER_WEEKLY_AD_SPEND_GID"
   PRODUCTS_GID: str = "PLACEHOLDER_PRODUCTS_GID"
   # ... 13 more placeholders requiring manual population
   ```

2. **Unresolved Schema Sources** (`/src/autom8_asana/dataframes/schemas/unit.py`):
   ```python
   ColumnDef(
       name="mrr",
       dtype="Decimal",
       source=None,  # Custom field GID - populated in Phase 3
   )
   ```

3. **Scale Problem**: 16 static GIDs for 2 task types (Unit, Contact). The legacy system has 50+ task types. Static GIDs do not scale.

**Why This Is Problematic**:

| Issue | Impact | Scale Factor |
|-------|--------|--------------|
| Manual GID maintenance | Each new field requires discovering and hardcoding a GID | O(n) per field |
| Environment coupling | Dev/staging/prod workspaces have different GIDs for the same fields | 3x maintenance |
| No workspace portability | SDK cannot be used across different Asana workspaces without code changes | Blocks reuse |
| Tight coupling | Schema definitions depend on specific workspace configurations | Brittle |
| Onboarding friction | New developers must learn which GIDs map to which fields | Hidden knowledge |

**Impact of Not Solving**:
- Every new custom field requires a code change and deployment
- Cannot share SDK across multiple Asana workspaces (each has unique GIDs)
- Testing requires maintaining separate GID mappings per environment
- 50+ task types would require 200+ hardcoded GIDs
- Field renames in Asana require no action (GIDs stable), but initial discovery is entirely manual

## Scope

### In Scope

- **Name Normalization Protocol**: Define canonical rules for matching schema field names to Asana custom field names
- **Resolution Algorithm**: Define how and when field names are resolved to GIDs
- **Resolution Caching**: Cache name-to-GID mappings for performance
- **Ambiguity Handling**: Define behavior when multiple fields could match
- **Missing Field Handling**: Define behavior when schema field has no Asana match
- **Testing Support**: Enable testing without live Asana connection
- **Explicit Override**: Preserve ability to specify GID directly via `ColumnDef.source`

### Out of Scope

- **Custom field creation**: This PRD does not cover creating custom fields in Asana
- **Field type validation**: Verifying that resolved field types match schema expectations (covered by FR-MODEL-005 in PRD-0003)
- **Multi-workspace simultaneous use**: Using multiple workspaces in a single SDK session
- **Custom field value updates**: Writing values back to Asana custom fields
- **Historical GID tracking**: Tracking when fields are renamed in Asana
- **Persistent cross-session cache**: Redis-based resolution cache (Could for future)

## Requirements

### Name Normalization Requirements (FR-NORM-*)

| ID | Requirement | Priority | Acceptance Criteria |
|----|-------------|----------|---------------------|
| FR-NORM-001 | SDK shall normalize schema field names from snake_case to a canonical form for matching | Must | `normalize("weekly_ad_spend")` returns consistent canonical form; verified by unit test |
| FR-NORM-002 | SDK shall normalize Asana custom field names to the same canonical form | Must | `normalize("Weekly Ad Spend")`, `normalize("weekly_ad_spend")`, `normalize("WeeklyAdSpend")` all return identical canonical form |
| FR-NORM-003 | Normalization shall be case-insensitive | Must | `normalize("MRR") == normalize("mrr") == normalize("Mrr")` verified by unit test |
| FR-NORM-004 | Normalization shall treat underscores, spaces, and absent separators equivalently | Must | `normalize("weekly_ad_spend") == normalize("Weekly Ad Spend") == normalize("WeeklyAdSpend")` |
| FR-NORM-005 | Normalization shall preserve acronyms as single tokens | Should | `normalize("MRR")` treats "MRR" as one token, not three letters |
| FR-NORM-006 | Normalization rules shall be documented with a truth table | Must | Documentation includes table mapping 20+ input variations to canonical forms |
| FR-NORM-007 | SDK shall expose a public `normalize_field_name()` function for debugging | Should | Function accessible from `autom8_asana.dataframes` namespace; returns canonical form |

### Resolution Algorithm Requirements (FR-RESOLVE-*)

| ID | Requirement | Priority | Acceptance Criteria |
|----|-------------|----------|---------------------|
| FR-RESOLVE-001 | SDK shall resolve schema field names to custom field GIDs at extraction time | Must | First task extraction triggers resolution; verified by mock that tracks resolution calls |
| FR-RESOLVE-002 | Resolution shall use the task's `custom_fields` list as the source of truth | Must | No additional Asana API calls made during resolution; verified by API call counter |
| FR-RESOLVE-003 | SDK shall match normalized schema field name against normalized custom field names | Must | Schema field `mrr` matches task custom field named "MRR" or "Mrr" |
| FR-RESOLVE-004 | SDK shall store resolved GID in a session-scoped mapping cache | Must | Second extraction of same schema field uses cached GID; verified by resolution call count |
| FR-RESOLVE-005 | SDK shall support explicit GID override via `ColumnDef.source` | Must | `ColumnDef(name="mrr", source="1234567890")` uses GID `1234567890` without resolution |
| FR-RESOLVE-006 | SDK shall resolve all schema custom fields on first task encounter | Should | Single resolution pass for all 11 Unit fields; batch verified by test |
| FR-RESOLVE-007 | Resolution shall occur once per schema per session | Must | Creating 100 DataFrames reuses same resolution; verified by resolution call count = 1 |

### Caching Requirements (FR-CACHE-*)

| ID | Requirement | Priority | Acceptance Criteria |
|----|-------------|----------|---------------------|
| FR-CACHE-001 | SDK shall cache resolved name-to-GID mappings in memory | Must | Cache implemented; verified by `resolver.get_cache()` returning dict |
| FR-CACHE-002 | Cache shall be session-scoped (lifetime of SDK client instance) | Must | New client instance starts with empty cache; verified by test |
| FR-CACHE-003 | Cache shall be thread-safe for concurrent extraction | Must | 10 concurrent extractions produce consistent results; verified by threading test |
| FR-CACHE-004 | SDK shall provide cache invalidation method | Should | `resolver.clear_cache()` empties cache; next extraction re-resolves |
| FR-CACHE-005 | SDK shall support optional persistent cache (Redis) | Could | Post-implementation: Share resolution across processes |
| FR-CACHE-006 | Cache shall store resolution metadata for debugging | Should | `resolver.get_resolution_info(field)` returns `{gid, resolved_at, source_field_name}` |

### Ambiguity Handling Requirements (FR-AMBIG-*)

| ID | Requirement | Priority | Acceptance Criteria |
|----|-------------|----------|---------------------|
| FR-AMBIG-001 | SDK shall detect when multiple custom fields match a schema field name | Must | Task with fields "MRR" and "mrr" triggers ambiguity detection for schema field `mrr` |
| FR-AMBIG-002 | SDK shall raise `AmbiguousFieldError` with all matching field names and GIDs | Must | Error message includes: field name, list of candidates with GIDs |
| FR-AMBIG-003 | SDK shall support disambiguation via explicit GID in `ColumnDef.source` | Must | Ambiguous field with explicit `source` does not raise error |
| FR-AMBIG-004 | SDK shall log warning when field match requires normalization (not exact) | Should | Log entry when "Weekly Ad Spend" matches `weekly_ad_spend` |
| FR-AMBIG-005 | SDK shall prefer exact case-sensitive match over normalized match | Must | Schema `MRR` prefers custom field "MRR" over "mrr" when both exist |

### Missing Field Handling Requirements (FR-MISSING-*)

| ID | Requirement | Priority | Acceptance Criteria |
|----|-------------|----------|---------------------|
| FR-MISSING-001 | SDK shall handle schema fields with no matching custom field gracefully | Must | Extraction continues; unresolved field value is `None`; no exception raised |
| FR-MISSING-002 | SDK shall log warning when schema field cannot be resolved | Must | Warning log includes: schema field name, list of available custom field names |
| FR-MISSING-003 | SDK shall support strict mode via `strict=True` parameter | Should | `to_dataframe(strict=True)` raises `FieldNotFoundError` on unresolved field |
| FR-MISSING-004 | SDK shall track which fields failed resolution | Must | `resolver.get_unresolved_fields()` returns `list[str]` of field names |
| FR-MISSING-005 | SDK shall distinguish "field not on task" from "field not in workspace" | Should | Different log messages: "Field X not present on task Y" vs "Field X not found in custom fields" |

### Testing Support Requirements (FR-TEST-*)

| ID | Requirement | Priority | Acceptance Criteria |
|----|-------------|----------|---------------------|
| FR-TEST-001 | SDK shall support mock resolver injection for testing | Must | Test can inject `MockResolver(mappings={"mrr": "123"})` |
| FR-TEST-002 | SDK shall provide factory for creating test fixtures with resolved schemas | Should | `create_resolved_schema(UnitSchema, {"mrr": "123", "discount": "456"})` |
| FR-TEST-003 | SDK shall support resolution from fixture data without Asana API | Must | Tests with fixture tasks work offline; no network calls |
| FR-TEST-004 | SDK shall provide `FailingResolver` for error path testing | Should | `FailingResolver(fail_on=["mrr"])` raises on specific fields |

## User Stories / Use Cases

### US-1: Developer Adds New Custom Field to Schema

As a developer, I want to add a new custom field to a schema without looking up GIDs so that I can quickly extend the data model.

**Scenario**:
1. Developer needs to add `renewal_date` field to Unit schema
2. Developer adds `ColumnDef(name="renewal_date", dtype="Date", nullable=True)`
3. Developer runs extraction against project with "Renewal Date" custom field
4. SDK resolves `renewal_date` to Asana's "Renewal Date" field automatically
5. Extraction succeeds; `renewal_date` column populated

**Acceptance**: New field works without hardcoded GID; time-to-add-field < 5 minutes.

### US-2: SDK Used in Different Workspace

As a team lead, I want to use the same SDK in our staging workspace so that we can test data pipelines without affecting production.

**Scenario**:
1. Production workspace has custom field "MRR" with GID `1111111111111111`
2. Staging workspace has custom field "MRR" with GID `2222222222222222`
3. Same schema code deployed to both environments
4. SDK resolves to correct GID in each environment automatically
5. Both environments work without code changes

**Acceptance**: Same code works in both workspaces; zero environment-specific GID configuration.

### US-3: Debugging Resolution Failures

As a developer, I want to see which GIDs were resolved so that I can debug field mapping issues.

**Scenario**:
1. Extraction returns unexpected `None` values for `discount` field
2. Developer calls `resolver.get_resolution_info("discount")`
3. Output shows: `{"status": "unresolved", "reason": "no match", "available_fields": ["Discount %", "MRR", ...]}`
4. Developer realizes Asana field is "Discount %" not "Discount"
5. Developer updates schema to `name="discount_percent"` or adds explicit `source`

**Acceptance**: Resolution debugging takes < 5 minutes; clear error messages.

### US-4: Handling Ambiguous Field Names

As a developer, I want clear errors when field names are ambiguous so that I can resolve conflicts explicitly.

**Scenario**:
1. Asana workspace has both "MRR" (number) and "mrr" (text) custom fields
2. Schema defines `ColumnDef(name="mrr", dtype="Decimal")`
3. Extraction raises `AmbiguousFieldError`
4. Error message: "Field 'mrr' matches multiple custom fields: 'MRR' (gid=111), 'mrr' (gid=222). Use explicit source to disambiguate."
5. Developer updates to `ColumnDef(name="mrr", source="111", dtype="Decimal")`

**Acceptance**: Ambiguity detected and reported with actionable resolution steps.

## Assumptions

| Assumption | Basis | Risk if Wrong |
|------------|-------|---------------|
| Task's `custom_fields` list contains all project custom fields | Asana API returns custom fields when `opt_fields=custom_fields` requested | Would need project-level API call |
| Custom field names are reasonably consistent within a workspace | Admin-controlled naming; UI encourages consistency | Lower resolution success rate |
| Most field names follow Title Case or snake_case conventions | Asana UI defaults; common practice | Normalization rules may need expansion |
| GIDs are stable and never change | Asana API guarantee (per ADR-0030) | Would invalidate entire caching approach |
| Single workspace per SDK session | Current SDK design | Multi-workspace would need per-workspace cache |
| First task in extraction has representative custom fields | All tasks in project have same custom field structure | May need multiple tasks for complete resolution |

## Appendix A: Normalization Truth Table

The following table defines expected normalization behavior. All inputs in a row should produce the same canonical output.

| Input Variations | Canonical Output | Notes |
|------------------|------------------|-------|
| `mrr`, `MRR`, `Mrr` | `mrr` | Acronym preserved as single token |
| `weekly_ad_spend`, `Weekly Ad Spend`, `WeeklyAdSpend`, `weekly ad spend` | `weekly_ad_spend` | Separators normalized |
| `discount`, `Discount`, `DISCOUNT` | `discount` | Simple case normalization |
| `office_phone`, `Office Phone`, `OfficePhone` | `office_phone` | CamelCase split |
| `vertical_id`, `Vertical ID`, `VerticalID`, `Vertical Id` | `vertical_id` | ID as suffix |
| `contact_email`, `Contact Email`, `ContactEmail` | `contact_email` | Standard compound |
| `full_name`, `Full Name`, `FullName` | `full_name` | Standard compound |

## Appendix B: Migration from Static GIDs

This PRD enables removing the static GID approach from ADR-0030. Migration steps:

1. **Implement resolver** per this PRD
2. **Update schemas** to remove `source=None` comments; field names become the contract
3. **Delete `custom_fields.py`** placeholder constants
4. **Update ADR-0030** status to "Superseded by PRD-0003.1"
5. **Verify tests** pass with dynamic resolution

Post-migration, adding a new custom field to a schema requires only:
```python
# Before (static GIDs):
# 1. Look up GID in Asana
# 2. Add constant to custom_fields.py
# 3. Add ColumnDef with source=CONSTANT

# After (dynamic resolution):
ColumnDef(name="new_field", dtype="Utf8", nullable=True)
# That's it. Resolution happens automatically.
```
