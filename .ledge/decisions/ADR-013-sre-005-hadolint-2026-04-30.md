# ADR-013 — Dockerfile lint enforcement via hadolint + custom M-16 assertion (SRE-005)

## §1 Context

VERDICT-eunomia-final-adjudication-2026-04-29 §7 routed M-16 (Dockerfile pattern
enforcement) to /sre as a design decision: choose a tool to enforce the
Dockerfile hygiene patterns codified in INVENTORY-pipelines §8.1, with a
watch-trigger of 2026-06-30 for orphan-risk reactivation.

The empirical M-16 surface is 7 patterns:

- **P-1**: Stage-0 `FROM ... AS secrets-extension` exists
- **P-2**: Stage-2 `COPY --link --from=secrets-extension`
- **P-3**: All cross-stage COPY use `--link`
- **P-4**: Base images SHA-pinned (not tag-only)
- **P-5**: Multi-stage discipline (builder/runtime separation)
- **P-6**: Non-root `USER appuser` UID 1000
- **P-7**: HEALTHCHECK present

P-1/P-2/P-3 are the watch-trigger core (cultural-only at PR #34); P-4..P-7 are
adjacent hygiene. The autom8y-asana repo carries 2 Dockerfiles:

| # | Path | Class |
|---|---|---|
| 1 | `Dockerfile` (145 LOC, 3-stage) | PRODUCTION (ECS+Lambda dual-mode, FR-CICD-001) |
| 2 | `Dockerfile.dev` (41 LOC, 1-stage) | DEV (volume-mount, TDD-LOCAL-DEV-ENV §4.3) |

Sprint-3 charter §5 dispatched B1 (audit) + B2 (integration) as separate atomic
sub-sprints. B1 (this audit at `.sos/wip/sre/AUDIT-sre-005-dockerfile-2026-04-30.md`)
adjudicated Q2(a) HADOLINT readiness with structural caveat: hadolint covers
adjacent P-4/P-6 STRONG and 4/7 patterns directly, but watch-trigger core
P-1/P-2/P-3 requires custom assertion (P-1 stage-name semantics out-of-scope;
P-2 PARTIAL via DL3022; P-3 buildkit feature lacks a stable rule).

## §2 Decision

**Adopt a layered Dockerfile lint architecture**:

1. **`hadolint`** for general Dockerfile hygiene (P-4/P-6 STRONG + broad
   DL-corpus catches for apt-get/USER/CMD/ENV patterns), driven by
   `.hadolint.yaml` config at repo root.
2. **Custom bash assertion** for M-16 watch-trigger core P-1/P-2/P-3
   (stage-name + `COPY --link` patterns hadolint does not directly cover).

Both layers run as separate jobs in a NEW workflow
`.github/workflows/dockerfile-lint.yml`, path-filtered to `Dockerfile*` +
`.hadolint.yaml` + the workflow file itself. Pre-commit hook entry
(`hadolint-docker`, already present at v2.12.0) refreshed to v2.14.0 per B1 §5
maintenance probe.

Authority chain:
- User dispatch 2026-04-30T18:31Z → Sprint-3 charter §1.1 + §5.1 Q2(a)
- B1 audit §4 + §8 (Q2(a) CONFIRMED with custom-assertion caveat)
- This ADR

## §3 Configuration

### §3.1 `.hadolint.yaml` (repo root)

```yaml
failure-threshold: warning
trustedRegistries: [public.ecr.aws, ghcr.io, docker.io]
override:
  error: [DL3007, DL3002, DL3022, DL3025]
  warning: [DL3006, DL3009, DL3015, DL3042, DL3059]
ignored: [DL3008]
```

Rationale per B1 §6:
- Error tier: P-4 (DL3007 latest-tag), P-6 (DL3002 USER-not-root), P-2 partial
  (DL3022 COPY-from), JSON CMD/ENTRYPOINT (DL3025) — STRONG-coverage of M-16
  adjacent surface.
- Warning tier: apt-get hygiene + pip-cache + RUN-merge — non-blocking for
  Dockerfile.dev intentional unpins.
- DL3008 ignored: documented dev-class carve-out (TDD-LOCAL-DEV-ENV §4.3
  Dockerfile.dev curl/gcc unpinned by design).

### §3.2 `.github/workflows/dockerfile-lint.yml`

Two jobs:

- **`hadolint`** matrix `[Dockerfile, Dockerfile.dev]` using
  `hadolint/hadolint-action@2332a7b74a6de0dda2e2221d575162eba76ba5e5 # v3.3.0`
  (B1 §5 SHA-pin candidate) + `.hadolint.yaml` + `failure-threshold: warning`.
- **`m16-pattern-assert`** bash grep for P-1/P-2/P-3:
  - P-1: `grep -qE '^FROM .* AS secrets-extension$' Dockerfile`
  - P-2: `grep -qE 'COPY --link --from=secrets-extension' Dockerfile`
  - P-3: negation check — any `^COPY [^-]` (bare COPY without flag prefix) in
    Dockerfile fails the job. Dockerfile.dev exempt by design (single-stage).

`on.push.branches: [main]` + `on.pull_request.branches: [main]` both
path-filtered to `Dockerfile*` / `.hadolint.yaml` / the workflow file.
`permissions: contents: read` minimal.

### §3.3 `.pre-commit-config.yaml` hook refresh

Existing `hadolint-docker` entry at `rev: v2.12.0` bumped to `rev: v2.14.0` to
match B1 §5 maintenance state (released 2025-09-22, ~7mo at audit time;
upstream commits 2026-04-30 confirm HEALTHY). Args extended with
`--config .hadolint.yaml --failure-threshold warning` for parity with CI.

## §4 Alternatives Rejected

### §4.1 Custom-grep-only (no hadolint)

**Rejected**: misses STRONG-coverage of P-4/P-6 (DL3007/DL3002) and the broader
DL-corpus surface (apt-get hygiene, JSON CMD, etc.). Custom grep would replicate
hadolint's 70+ rule engine in bash — prohibitive maintenance burden for
strictly worse coverage.

### §4.2 Docker buildx + `dive` image-altitude scanning

**Rejected**: image-altitude not Dockerfile-altitude. M-16 patterns are
Dockerfile-source semantics (stage-naming, COPY-flag discipline) that disappear
once the image is built. dive scans a built image's layers for size/efficiency,
not source patterns. Wrong altitude.

### §4.3 Defer indefinitely

**Rejected**: VERDICT-eunomia §7 watch-trigger 2026-06-30 (~2 months from
audit date) risks the M-16 surface becoming orphan if no enforcement primitive
exists by the trigger date. Charter §5 explicitly scoped B1+B2 as the
discharge path.

### §4.4 hadolint-only (no custom layer)

**Rejected**: B1 §4 coverage matrix shows P-1 stage-name semantics + P-3
`--link` enforcement are 0% covered by hadolint's DL-corpus. Without the custom
layer, M-16 watch-trigger core regression is undetectable. Layered approach is
the standard industry pattern (general lint + project-specific assertion).

## §5 Risks

### §5.1 hadolint maintainer-velocity (charter §11.3)

**Severity**: LOW. **Mitigation**: SHA-pin + manual refresh cadence. B1 §5
probed upstream 2026-04-30: HEALTHY (v2.14.0 released 7mo ago, commits TODAY,
12,099 stars, 241 open issues actively triaged). `hadolint-action` v3.3.0
released alongside.

**Decay signal**: if upstream `pushed_at` exceeds 18 months without release,
re-probe per charter §5.2 HALT trigger criteria. Surface as DRIFT-FINDING.

### §5.2 Custom-assertion grep brittleness

**Severity**: LOW. **Mitigation**: regex anchored to start-of-line (`^FROM`,
`^COPY`), with explicit `[^-]` exclusion for flag-prefixed forms. Today's
Dockerfile passes all three assertions (verified 2026-04-30 via direct grep
probe at audit-substrate file-read time). Future Dockerfile evolution that
intentionally restructures these stages will fail the gate — by design;
expectation is ADR amendment, not silent assertion drift.

### §5.3 Dockerfile.dev future drift to prod

**Severity**: MODERATE. **Mitigation**: `ignored: [DL3008]` carries inline
comment "Lift if dev → prod migration occurs." If TDD-LOCAL-DEV-ENV §4.3
contract changes (Dockerfile.dev becomes prod-bound), this ADR requires
amendment to remove the carve-out and pin curl/gcc versions.
