## §5 Mutation Pattern (per-group Edit shape)

For each group G_n, janitor performs ONE Edit replacing the contiguous rejection-test block with a single parametrized test function. Pattern:

```python
@pytest.mark.parametrize(
    ("field", "value", "must_contain"),
    [
        ("max_requests", 0, ("max_requests", "positive")),
        ("max_requests", -10, ("max_requests",)),
        ("window_seconds", 0, ("window_seconds", "positive")),
        ("window_seconds", -5, ("window_seconds",)),
    ],
    ids=[
        "rejects_zero_max_requests",
        "rejects_negative_max_requests",
        "rejects_zero_window_seconds",
        "rejects_negative_window_seconds",
    ],
)
def test_rejects_invalid_field(
    self, field: str, value: int | float, must_contain: tuple[str, ...]
) -> None:
    """Rejects invalid value for the named field."""
    with pytest.raises(ConfigurationError) as exc_info:
        RateLimitConfig(**{field: value})
    for token in must_contain:
        assert token in str(exc_info.value)
```

**Mutation invariants**:
- Constructor invocation uses `**{field: value}` kwargs unpack — preserves single-field-under-test isolation.
- `ids=` list mirrors the original test function names verbatim — preserves pytest collection-line readability + `-k` filter compatibility.
- `must_contain` is a tuple iterated via `for token in must_contain: assert token in ...` — preserves multi-substring specificity (rows with 2 substrings remain 2-assertion).
- Numeric type union `int | float` covers TimeoutConfig float values + integer rows.

**Per-group Edit blocks**:
- **G1 (RateLimitConfig)**: Edit replaces L43–L71 (4 tests, ~29 lines) with 1 parametrized block (~22 lines). Net: ~−7 lines.
- **G2 (RetryConfig parametrize-eligible)**: Edit replaces L110–L145 rejection block (rows 5–8, interleaved with `accepts_*` tests at L118–L122 → janitor preserves intermixed `accepts_*` tests; parametrize block placed AFTER the last interleaved `accepts_*`). Net: ~−10 lines.

- **G4 (TimeoutConfig)**: Edit replaces L236–L294 (8 tests). Net: ~−25 lines.
- **G5 (ConnectionPoolConfig)**: Edit replaces L320–L363. Net: ~−18 lines.

## §6 Assertion-Specificity-Preservation Rules (HANDOFF AC 5 hard gate)

R1 — **Per-case substring tuple is mandatory**: every parametrize row carries the exact substrings the original test asserted; the parametrized assertion loop iterates the tuple. NO generic catch-all (e.g., `assert "Error" in str(...)`).

R2 — **Exception subclass binding**: every parametrize case asserts against `ConfigurationError` (NOT bare `Exception` / `ValueError`). Uniform across all 27 source tests; no relaxation.

R3 — **Structural outlier escape valve**: any source test whose assertion substring set is structurally non-conforming to its peers' set (e.g., row 9's "at least 1" vs G2's "non-negative"/"positive" pattern) is RETAINED as standalone (PARAMETRIZE-PARTIAL-CLOSE per charter §6.3). Decision rule: if folding requires asymmetric branch logic in the parametrized body, RETAIN.

R4 — **Test-id preservation**: pytest collection IDs (via `ids=`) match original function names verbatim, so `pytest -k "rejects_zero_max_requests"` continues to resolve.

R5 — **No assertion-side relaxation**: substring-match semantics (`"foo" in str(exc)`) are preserved verbatim. Janitor MUST NOT "tighten" to `==` or "loosen" to regex.

R6 — **Audit-lead 3-case specificity sample**: per charter §6.2, audit-lead samples 3 of the 32 collected cases and confirms the original assertion-substring set is present. Janitor surfaces sample candidates: (G1 row 1 zero/positive 2-substring), (G2 row 5 negative/non-negative 2-substring), (G4 row 21 negative/single-substring).
