## 2. Data Shape Inventory

### Pipeline Data Flow

```
DataServiceClient API calls
    |
    v
InsightsResponse.data: list[dict[str, Any]]    # Raw row dicts
    |
    v
TableResult(table_name, success, data, row_count)  # Per-table wrapper
    |
    v
InsightsReportData(business_name, phone, vertical,
                   table_results: dict[str, TableResult])  # Full report input
    |
    v
compose_report() -> str                        # Formatted document string
    |
    v
str.encode("utf-8") -> bytes                   # For upload
    |
    v
AttachmentsClient.upload_async(file=bytes, name=str, content_type=str)
```

### Table Row Shape (from API)

Each table's `data` field is `list[dict[str, Any]]`. Column names are snake_case strings. Values are mixed types:

| Type | Examples | Handling |
|------|----------|----------|
| `str` | business names, dates, phone numbers | Direct string rendering |
| `int` | counts (impressions, clicks, leads) | `str(value)` |
| `float` | currency (spend), percentages | `str(value)` -- no formatting applied |
| `None` | missing data | Rendered as `"---"` |

**Important**: The formatter does NOT perform numeric formatting (no thousands separators, no currency symbols, no percentage signs). Values are rendered as-is from the API. An HTML formatter could add `text-align: right` for numeric columns, which would be an improvement.

### Column Discovery

The formatter discovers columns dynamically from row keys (lines 162-168):
```python
columns: list[str] = []
seen: set[str] = set()
for row in display_rows:
    for key in row:
        if key not in seen:
            columns.append(key)
            seen.add(key)
```

This means the formatter handles heterogeneous rows (different rows may have different keys). An HTML formatter must preserve this behavior.

## 4. Existing Patterns

### 4.1 Templating

**No template engine exists in the codebase.** Searches for Jinja2, Mako, or any template library returned zero results in both `pyproject.toml` dependencies and source code imports.

The `automation/templates.py` file is a **TemplateDiscovery** class for finding Asana template tasks -- completely unrelated to document templating.

### 4.2 HTML Generation

**No HTML generation exists anywhere in the codebase.** The term "html" appears only in docstrings, test data, and error messages -- never as an output format.

### 4.3 Report Composition Patterns

The current pattern is a single `compose_report()` function that concatenates sections:
```python
def compose_report(data: InsightsReportData) -> str:
    sections: list[str] = []
    sections.append(_format_header(data))
    for table_name in TABLE_ORDER:
        # ... dispatch to table/empty/error formatters
    sections.append(_format_footer(...))
    return "\n\n".join(sections) + "\n"
```

This is a pure-function pipeline with no class hierarchy, no protocol, and no formatter abstraction. The `compose_report -> str` interface is the only contract consumed by `insights_export.py`.

### 4.4 Configuration for Output Format

**No format selection mechanism exists.** The format is hardcoded:
- Extension: `.md` hardcoded in f-string (line 554)
- Content type: `"text/markdown"` literal (line 561)
- Attachment pattern: `"insights_export_*.md"` constant (line 54) and lambda handler default (lambda line 42)

There is no enum, config flag, or environment variable for format selection.

### 6.5 Asana API Content-Type Restrictions

**None.** The Asana attachment upload API (`POST /tasks/{gid}/attachments`) accepts any file with any MIME type via multipart/form-data. The `content_type` field is informational metadata -- Asana does not validate or reject based on it. Both `text/html` and `text/plain` are accepted.

### 6.6 Transitional Cleanup Risk

When changing from `.md` to `.html`, old `.md` attachments on Asana tasks will NOT be cleaned up by the new `.html` pattern. The cleanup mixin uses `fnmatch.fnmatch(att_name, pattern)` -- `"insights_export_*.html"` will not match `"insights_export_foo_20260219.md"`.

**Mitigation**: The first deployment must temporarily include both patterns. The mixin currently accepts a single pattern string. Options:
1. Pass the old pattern as a one-time override via Lambda event: `{"attachment_pattern": "insights_export_*.md"}` on a single execution
2. Extend the mixin to accept multiple patterns (more invasive)
3. Accept stale `.md` files until manual cleanup

Recommended: Option 1 -- run one execution with old pattern after deploying the new code, then switch to the new pattern. This requires zero code changes to the mixin.

`src/autom8_asana/api/lifespan.py:210-214` imports and registers the lambda handler config for API invoke. This is an indirect dependency on `_config.default_params["attachment_pattern"]` which must be updated.

**Implementation sketch for `_format_table_section()`**:
```python
def _format_table_section(
    table_name: str,
    rows: list[dict[str, Any]],
    row_limit: int | None = None,
) -> str:
    total_rows = len(rows)
    display_rows = rows[:row_limit] if row_limit else rows
    truncated = row_limit is not None and total_rows > row_limit

    columns = _discover_columns(display_rows)
    if not columns:
        return f'<section id="{_slugify(table_name)}"><h2>{table_name}</h2><p class="empty">No data available</p></section>'

    header_cells = "".join(f"<th>{_to_title_case(c)}</th>" for c in columns)
    body_rows = []
    for row in display_rows:
        cells = "".join(f"<td>{_format_cell_html(row.get(c))}</td>" for c in columns)
        body_rows.append(f"<tr>{cells}</tr>")

    parts = [
        f'<section id="{_slugify(table_name)}">',
        f'<h2>{table_name} <span class="badge">{len(display_rows)}</span></h2>',
        '<div class="table-wrap"><table>',
        f'<thead><tr>{header_cells}</tr></thead>',
        f'<tbody>{"".join(body_rows)}</tbody>',
        '</table></div>',
    ]
    if truncated:
        parts.append(f'<p class="truncation">Showing first {row_limit} of {total_rows} rows</p>')
    parts.append('</section>')
    return "\n".join(parts)
```

| Keep + refactor markdown formatter | 0.5 day | Medium |
| Config/selection mechanism | 0.5 day | Medium |
| Full test suite (both formatters + protocol) | 1-1.5 days | Medium |
| Integration + transitional cleanup | 0.5 day | Medium |

**Confidence**: Medium (6/10). The protocol itself is simple, but the question is: **do we need the markdown formatter going forward?** The spike concluded markdown is "strictly worse than every alternative." Keeping a dead code path adds maintenance burden for no value.

**Assumptions**:
- There is a future need for multiple output formats (not currently established)
- The markdown formatter will be maintained alongside HTML (unclear)
- Configuration complexity is acceptable for the team

**Risks**:
- Over-engineering for a single format change
- Dead code path (markdown formatter) if never used
- Configuration sprawl (env var + lambda event + API parameter)

**Rollback path**: Same as Option A (git revert), plus remove the protocol and revert to direct formatter.


### Option C: HTML Replacement with `tabulate` Bridge (Hybrid)

**Approach**: Add `tabulate` dependency, use `tablefmt="html"` for table markup, wrap in an HTML document template. This is the "escalation path" mentioned in the spike report.

**Changes required**: Similar to Option A, but table generation uses `tabulate(rows, headers="keys", tablefmt="html")` instead of hand-crafted HTML.

**Effort estimate**: 2 days

**Confidence**: Medium (7/10). `tabulate` HTML output is functional but limited -- it produces bare `<table>` tags without CSS classes, IDs, or data attributes. All styling must be done via tag-based CSS selectors. The POC's interactive features (search, sort, collapse) cannot be built on `tabulate` output without post-processing.

**Trade-off**: Saves ~50 lines of table generation code but adds a dependency and constrains future table enhancements.

**Risks**:
- `tabulate` HTML output lacks `class`/`id` attributes for CSS targeting
- No control over `<thead>`/`<tbody>` separation (tabulate does not emit them for HTML)
- Post-processing tabulate output to add attributes defeats the purpose

**Rollback path**: Same as Option A.

## 8. Recommendation

**Recommended: Option A (In-Place HTML Replacement)**

### Rationale

1. **Minimal change surface.** The formatter is an isolated 274-line pure-function module. Replacing its output format is a clean, contained change. The `compose_report() -> str` contract does not change -- only what's inside the string changes.

2. **No new dependencies needed.** HTML generation is string concatenation with f-strings, exactly like the current markdown generation. Adding `tabulate` or Jinja2 introduces dependencies for a problem that doesn't need them.

3. **Stakeholder approved the POC.** The HTML approach was validated with real production data. The question is not "should we use HTML?" but "how do we wire it in?"

4. **Option B is premature abstraction.** There is no evidence of a second output format need. The spike explicitly stated: "CSV is the correct format for conversation_audit (single table, opens in Excel). No change needed there." A formatter protocol would serve exactly one implementation.

5. **Option C constrains future enhancement.** The POC demonstrated 12 interactive features (search, sort, dark mode, KPI cards, collapsible sections). Using `tabulate` for table markup forecloses most of these without post-processing. Hand-crafted HTML gives full control.

6. **Test rewrite is mechanical.** The 38 formatter tests assert on specific string patterns. They need updating regardless of which option we choose. The effort is the same.

### Why Not tabulate?

The spike's original recommendation was `tabulate` + `.txt` for plain text. The stakeholder then requested the HTML POC, which was built and approved. The HTML approach supersedes the tabulate recommendation. Using `tabulate(tablefmt="html")` produces inferior HTML compared to direct generation:

- No `<thead>`/`<tbody>` separation
- No `class` attributes for CSS targeting
- No `id` attributes for navigation anchors
- No row-count badges, truncation indicators, or error markers
- Post-processing tabulate output to add these defeats the purpose

Direct HTML generation is ~50 more lines of code but provides full control over the markup structure.
