### 1.2 Architectural Decisions

**Decision 1 (SM-001 sequencing): Batch by risk, 4 tiers.**

Converting all 16 client files simultaneously is too large a blast radius. Instead, batch by complexity:

- **Tier A (validation)**: `users.py` (simplest -- 3 methods, 289 lines, well-tested). Proves the conversion pattern works for the remaining clients.
- **Tier B (standard)**: `workspaces.py`, `teams.py`, `tags.py`, `webhooks.py`, `goals.py` -- simple CRUD clients with 1-6 sync pairs each.
- **Tier C (complex)**: `projects.py`, `stories.py`, `portfolios.py`, `attachments.py`, `custom_fields.py`, `goal_followers.py`, `goal_relationships.py`, `name_resolver.py` -- more methods and/or non-standard patterns.
- **Tier D (highest risk)**: `tasks.py`, `task_operations.py` -- largest files, most sync pairs (5+6=11), `task.model._client` reference pattern, `_resolve_entity_ttl` delegation.

The `batch/client.py`, `persistence/session.py`, `models/task.py`, and `models/business/seeder.py` also use `sync_wrapper` but are NOT Asana resource clients. These are **out of scope** for SM-001 -- they use `sync_wrapper` for different reasons (session management, model convenience methods) and should not be converted to `@async_method`.

**Decision 2 (SM-006 disposition): Dismiss as acceptable cost of type safety.**

The ~2,400 lines of `@overload` declarations are a mypy/IDE static analysis requirement. They cannot be eliminated by runtime mechanisms. The three possible approaches (accept, codegen, Protocol) all have significant tradeoffs:

- **Accept**: Zero risk, zero effort. Overloads are boilerplate but they serve a purpose.
- **Codegen**: Introduces build tooling dependency, maintenance burden, IDE confusion during development.
- **Protocol generic base**: Would require a `TypeVar`-based `GenericClient[T]` pattern that may not compose with `BaseClient` inheritance without significant refactoring. The return type for `raw=True` (always `dict`) vs `raw=False` (varies per client) makes generic typing awkward.

**Verdict**: SM-006 is dismissed. The overload declarations are preserved during SM-001 conversion (they are still needed with `@async_method`). The net line reduction from SM-001 comes entirely from eliminating the `get() -> _get_sync() -> get_async()` delegation chain.

**Decision 3 (SM-002 pattern): Template method base class.**

Composition/decorator was considered but rejected because:
- The S3 and Redis backends share the SAME protocol (CacheProvider) with SAME method signatures
- The shared scaffolding (degraded check, timing, metrics, error handling) wraps EVERY method identically
- Template method lets us define the scaffolding once in a base class and have backends implement only the backend-specific operations

The base class (`CacheBackendBase`) will:
1. Absorb `DegradedModeMixin` (which it already inherits from both)
2. Provide template methods for each protocol operation (e.g., `get()` calls `_do_get()`)
3. Move timing, degraded checks, metrics, and error handling into the template
4. Backends implement only abstract `_do_get()`, `_do_set()`, etc.
5. Move freshness stamp serialization into the base class (SM-007 fix)
6. Move init boilerplate into base `__init__` (SM-008 fix)

This preserves the existing `CacheProvider` protocol contract while eliminating ~600 lines of structural duplication.

**Decision 4 (SM-004 + SM-002 convergence): Treat separately.**

The DataServiceClient retry pattern and the cache backend scaffolding pattern are superficially similar (both involve resilience wrapping) but differ in critical ways:

- Cache backends: synchronous, degraded-mode state machine, timing + metrics + error-to-degraded
- DataServiceClient: async, circuit-breaker + retry-with-backoff, stale-cache-fallback

A shared abstraction would be over-engineered. SM-002 is solved by template method within the cache module. SM-004 is solved by extracting a private `_execute_with_retry` method within DataServiceClient.

**Decision 5 (SM-003 approach): Use `sync_wrapper` function, not `@async_method`.**

`@async_method` is a descriptor that requires `__set_name__` -- it injects `{name}_async()` and `{name}()` into the class. DataServiceClient already has explicit `get_insights_async()` and `get_insights()` methods with different signatures and docstrings. Using `@async_method` would require restructuring the API surface.

Instead, SM-003 should extract the repeated event-loop-check + `asyncio.run()` pattern into calls to the existing `sync_wrapper` function. This is the minimal-risk approach that eliminates duplication without changing the class API.

However: on closer inspection, `sync_wrapper` is a decorator factory (`@sync_wrapper("method_name")`) that wraps an async function into a sync function. It does not work as a simple callable. The DataServiceClient `get_insights()` is a regular sync method that manually does `asyncio.run(self.get_insights_async(...))`. The pattern difference is:

- Old clients: `@sync_wrapper("get_async") async def _get_sync(...)` -- decorator wraps async func
- DataServiceClient: `def get_insights(...): asyncio.run(self.get_insights_async(...))` -- manual inline

The cleanest fix is to extract a small utility function `_run_sync(coro)` as a private method or module-level helper that encapsulates the loop-check + asyncio.run pattern, and use it in `get_insights()` and `__exit__()`. This is lower risk than refactoring to use `sync_wrapper` decorator pattern.

**Decision 6 (Phase ordering): WS-3 first, then WS-1, then WS-2.**

Rationale:
- **WS-3 (SM-005)**: Smallest scope (1 file), lowest risk, no cross-file dependencies. Quick win that proves the pipeline works.
- **WS-1 (SM-001, SM-003, SM-004)**: Medium scope (16+ files) but highly mechanical -- the SectionsClient provides an exact template. SM-003 and SM-004 are DataServiceClient-only and can execute independently.
- **WS-2 (SM-002, SM-007, SM-008)**: Highest risk -- introduces a new base class into the cache backend hierarchy. Should execute last when the team has confidence in the pipeline.
